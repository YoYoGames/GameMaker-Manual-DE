Feder

Feather bietet eine intelligente code Vervollständigung und eine verbesserte syntax Überprüfung in Ihrem GML Code scripts, zusammen mit intelligenten Refactoring Optionen.

Diese Seite enthält Informationen über die folgenden Teile von Feather:

  • Basis-Optionen
  • Hervorhebung
  • Schweregrad der Meldung ("GMXXXX"-Regeln)
  • Benennungsregeln (für Assets, Variablen usw.)

Basis-Optionen

Die grundlegenden Federeinstellungen enthalten die folgenden Optionen:

  • Aktivieren Sie Feather: Aktivieren oder deaktivieren Sie hier Feather. Wenn Sie diese Option deaktivieren, sind auf IDE keine Feather-Funktionen verfügbar.
  • Verzögerung beim Parsen von Dokumenten: Die Zeit, die Feather wartet, um Ihre script zu überprüfen, nachdem Sie eine Änderung daran vorgenommen haben.
  • Kompakte Tooltips: Wenn Sie diese Option aktivieren, werden bestimmte Tooltip-Informationen entfernt, die angezeigt werden, wenn Sie den Mauszeiger über Schlüsselwörter bewegen, z. B. Parameterbeschreibungen für Funktionen, wodurch die Tooltips kleiner werden.
  • Die maximale Anzahl von Strukturfeldern, die durch Tooltips angezeigt werden: Wenn Sie den Mauszeiger über eine Strukturreferenz bewegen, zeigt ein Tooltip die Variablen innerhalb dieser Struktur an, bis zu der hier eingegebenen Anzahl.
  • Refactor references to asset name when asset is renamed in Asset Browser: Wenn Sie diese Option aktivieren, wird sichergestellt, dass bei der Umbenennung eines Assets in Ihrem Asset-Browser alle Verweise auf dieses Asset in Ihrem code ebenfalls bearbeitet werden, damit diese Verweise nicht verloren gehen.
  • Bevorzuge mehrzeiligen JSDoc-Kommentarstil bei der Erstellung aus Quick Fix: Wenn Sie JSDoc-Kommentare für eine Funktion über das Menü "Quick Fixes" erzeugen, wird die erzeugte JSDoc entweder ein- oder mehrzeilige Kommentare verwenden, je nach dieser Option.
  • Aktivieren Sie den Modus Strict Type: Wenn dies aktiviert ist, achtet Feather strenger darauf, dass Sie die richtigen Datentypen in Ihrem code verwenden.

Hervorhebung

In diesem Abschnitt finden Sie Einstellungen, mit denen Sie die Farben ändern können, mit denen ein Teil Ihrer code unterstrichen wird, um einen Fehler, eine Warnung oder einen Vorschlag zu markieren:

  • Fehlerfarbe: Die für Fehler verwendete Farbe.
  • Warnfarbe: Die Farbe, die für Warnungen verwendet wird.
  • Vorschlagsfarbe: Die für Vorschläge verwendete Farbe.

Schweregrad der Meldung

Dieser Abschnitt enthält alle Regeln, mit denen Feather Ihre code prüft, und Sie können den Schweregrad jeder Regel einstellen.

Profil

In dieser Dropdown-Box können Sie zwischen einer Reihe von Voreinstellungen für die GM*-Regeln wählen. Die folgenden Profile sind verfügbar:

  • Keine: Dieses Profil zeigt keine Fehlermeldungen an.
  • SyntaxFehler: Wenn Sie dieses Profil verwenden, werden syntax Fehler angezeigt, zusammen mit den Fehlern, die im Profil None angezeigt werden.
  • TypFehler: Dieses Profil zeigt zusätzlich zum Profil SyntaxErrors auch Typfehler an.
  • Alles: Dieses Profil zeigt alles, was Feather analysieren kann.

GM*-Regeln

Sie können eine Regel als "Fehler", "Warnung" oder "Vorschlag" festlegen, was sich auf die Art und Weise auswirkt, in der Ihnen die Regelverletzung code im Code Editor und im Fenster Feather Messages gemeldet wird.

Sie können eine Regel auf "Ignorieren" setzen, wodurch alle Verstöße gegen diese Regel ignoriert werden.

Alle Regeln, die mit "GM1000" beginnen, überprüfen Ihr syntax auf mögliche fatale Fehler, und die Regeln, die mit "GM2000" beginnen, sind Best-Practice-Regeln, die Ihnen helfen, Fehler in Ihrem Spiel zu vermeiden.

Regeln für die Namensgebung

In diesem Bereich können Sie Benennungsregeln für Assets, Variablen und verschiedene andere Teile von GML Code festlegen.

Die GM2017-Regel unter "Meldungsschweregrad" muss aktiviert sein, damit die Benennungsregeln wirksam werden. Standardmäßig ist sie deaktiviert.

Die Einstellungen in diesem Abschnitt sind wie folgt:

  • Identifikator-Blockliste: Dies ist eine durch Leerzeichen getrennte Liste von Bezeichnern, die bei allen Benennungsregeln ignoriert werden.

    Angenommen, Sie haben eine Regel zur Verwendung von obj_ als Präfix für Objekte, aber Sie möchten, dass Ihr Objektmanager und Ihr Netzwerk unverändert bleiben und das Präfix obj_ nicht verwenden.

    In diesem Fall würden Sie manager network in das Feld Indentifier Blocklist schreiben. Feather ignoriert diese Objekte und alle anderen Bezeichner (Variablen, Enums, Parameter usw.) mit denselben Namen.
  • <Bezeichner> Benennungsregel: Sie können Benennungsregeln für jede Art von Bezeichner in der Liste festlegen, z. B. für Assets, Makros, Enums, Funktionsparameter, Variablen usw.

    Jedes Dropdown-Menü für eine Benennungsregel bietet die folgenden Optionen:
    • Präfix: Text, der vor dem Namen des Bezeichners erscheinen soll, z. B. obj_ für Objekte, _ für lokale Variablen usw., je nach Ihren Vorlieben.
    • Suffix: Text, der nach dem Bezeichnernamen erscheinen soll.
    • Benennungsstil: Wählen Sie den Benennungsstil für den Bezeichner. Deine Optionen sind:
      • Unbegrenzt: Für diesen Bezeichnertyp wird keine Formatierung erzwungen.
      • GroßKleinschreibung: MeineGesundheit, ObjGrappleHook, MaxInventoryItemsAllowed, usw.
      • lowerCamelCase: meinGesundheit, objGrappleHook, maxInventoryItemsAllowed, usw.
      • ALL_UPPER: MY_HEALTH, OBJ_GRAPPLE_HOOK, MAX_INVENTORY_ITEMS_ALLOWED, usw.
      • all_lower: mein_gesundheitszustand, obj_grapple_hook, max_inventory_items_allowed, usw.
