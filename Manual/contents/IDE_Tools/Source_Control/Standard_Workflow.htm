Standard-Workflow

Während Sie an Ihrem Projekt arbeiten, werden Sie natürlich sprites erstellen, scripts bearbeiten, Zeitleisten löschen und was sonst noch so anfällt, wenn sich Ihr Projekt im Laufe der Zeit entwickelt. Dies sind alles Aktionen, die Sie verfolgen und zurückverfolgen wollen, falls etwas schief geht. Dies ist der Hauptgrund für die Verwendung der Versionskontrolle - damit Unfälle rückgängig gemacht werden können - und wir werden Sie kurz durch den grundlegenden Arbeitsablauf von Anfang an führen, damit Sie verstehen, wie dies funktioniert.

Beginnen Sie von vorne: Erstellen Sie ein neues Projekt GML und aktivieren Sie die Versionskontrolle über die Hauptoptionen. Dadurch erhalten wir Zugriff auf das Menü "Versionskontrolle" im oberen Bereich und wählen sofort " Projektarchiv erstellen". Wenn Sie sich die Raumressourcen ansehen, können Sie sehen, dass der Standardraum ein rotes Ausrufezeichen erhalten hat, was bedeutet, dass er gegenüber seinem letzten Zustand verändert wurde. Wenn wir andere Ressourcen im Projekt hätten, würden Sie sehen, dass sie alle das gleiche Symbol haben, wie im Bild unten gezeigt:

Wir haben nun die Möglichkeit, das leere Projekt so zu committen, wie es ist, oder es in einen Basiszustand zu versetzen. Für den Moment machen wir unsere erste Übergabe, also klicken wir auf Source Control > Commit Changes. Es öffnet sich nun ein neues Fenster, das in drei Bereiche unterteilt ist:

  • Gestufte Änderungen: Dies sind die Änderungen, die in den Quellcode übertragen werden. Dies wird wahrscheinlich leer sein, kann aber auch eine Änderung der Hauptoptionen anzeigen, was in Ordnung ist.
  • Unveränderte Änderungen: Hier werden die Dateien angezeigt, die sich geändert haben, denen wir aber noch nicht mitgeteilt haben, dass sie übertragen werden sollen ( Git ) - es handelt sich um "ausstehende" Änderungen.
  • Commit Message: Dies ist eine Notiz, die wir hinzufügen können, um die Änderungen zu erklären.

Klicken Sie zunächst auf die Schaltfläche Stage All, da wir alle Änderungen festschreiben wollen, und geben Sie etwas wie "First Commit!" in die Commit Message ein und klicken Sie auf Commit. Alle unsere Ressourcen sollten nun ein grünes Häkchen haben, d.h. es wurden keine Änderungen gefunden.

Bei der Arbeit in diesem Fenster ist zu beachten, dass den verschiedenen Staged- und Unstaged-Meldungen ein Symbol vorangestellt ist. Das verwendete Symbol variiert je nach Vorgang und Datei, wobei das folgende verwendet wird:

  • [...] - Stufenweiser Wechsel (war vorher [A])
  • [+] - Neu hinzugefügt
  • [->] - Umbenannt/verschoben
  • [-] - Entfernt
  • [M] - Unveränderte Änderungen
  • [?] - Unverfolgte Datei
  • [!] - Fehlende Datei

Unser Arbeitsprozess kann nun beginnen, und wir können mit der Erstellung unseres Projekts und dem Speichern und Übertragen von Änderungen beginnen, so wie wir es in den obigen Abschnitten getan haben.

Wenn Sie eine Weile gearbeitet haben, möchten Sie vielleicht nachsehen, was Sie zu einem bestimmten Zeitpunkt committed haben, und dazu öffnen Sie den Commit-Verlauf. Um das Verlaufsfenster zu öffnen, gehen Sie einfach auf Versionsverwaltung > Verlauf anzeigen, was ein weiteres Fenster mit drei Bereichen öffnet:

Der obere Bereich beschreibt die Commits, der mittlere Bereich den Kommentar zu dem ausgewählten Commit und der untere Bereich die Dateien, die in diesem Commit geändert wurden. Wenn Sie ein Diff-Tool eingerichtet haben, können Sie mit einem Doppelklick auf eine beliebige Datei in diesem Fenster das Diff-Tool öffnen und die Änderungen zwischen den Dateien sehen.
