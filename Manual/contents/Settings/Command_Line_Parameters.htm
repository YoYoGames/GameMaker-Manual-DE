Befehlszeilenparameter

Beim Erstellen von Spielen ist es wichtig, dass Sie die ausführbare Datei testen, um sicherzustellen, dass die endgültige Kompilierung korrekt ist und keine unvorhergesehenen Fehler oder Abweichungen aufweist. Die Erstellung einer ausführbaren Datei bedeutet jedoch, dass Sie keinen Zugriff auf die Debug-Konsole oder das compiler -Fenster haben und es daher schwierig sein kann, etwaige Probleme zu finden und zu beheben. Es gibt jedoch eine Lösung für dieses Problem, und zwar die Verwendung von Befehlszeilenparametern, um bestimmte Eigenschaften festzulegen, wenn die ausführbare Version des Spiels ausgeführt wird, wodurch Sie wiederum Ausgabeprotokolldateien erstellen und andere spezielle Situationen testen können.

HINWEIS Diese Funktion ist nur für das Standard-Windows-Ziel module verfügbar.

Damit dies funktioniert, müssen Sie zunächst eine Windows ausführbare Datei Ihres Spiels erstellen und diese auf Ihrem Testrechner installieren. Danach müssen Sie eine Verknüpfung zu dem Spiel auf dem Desktop erstellen. Hier fügen wir die Parameter ein, die an *.exe übergeben werden sollen (um eine Verknüpfung zu erstellen, klicken Sie mit der rechten Maustaste auf die ausführbare Datei des Spiels und wählen Sie Senden an > Desktop (Verknüpfung erstellen)).

Sobald Sie Ihre Verknüpfung erstellt haben, können Sie einen der folgenden Parameter im Zielpfad verwenden:

-noaudio: Dies schaltet den gesamten Ton in Ihrem Spiel aus.

-inawindow: Erzwingt den Start des Spiels in einem Fenster, auch wenn es im Vollbildmodus läuft.

-output <filename>: Sendet die Konsolenausgabe an den Dateinamen, einschließlich aller Debug-Meldungen, die Sie in Ihrer code base haben.

-debugoutput <filename>: Sendet die Konsolenausgabe an den Dateinamen, ohne benutzerdefinierte Debug-Meldungen, aber mit zusätzlichen Informationen von runner für Fehlerberichte.

-software: Zwingt das Spiel, Software-Vertex-Verarbeitung anstelle von Hardware für das Rendern der Spielgrafik zu verwenden. Dies ist besonders nützlich für PCs, die einen integrierten Intel-GFX-Chipsatz verwenden oder auf älteren Rechnern Darstellungsprobleme aufweisen.

-intel: Dies wird switch auf einem Fix für schlechte Intel-Treiber, unabhängig von der GPU verwendet wird. Dieser Workaround für diese schlechten Treiber wird die Dinge verlangsamen, was zu einer spürbaren Leistungseinbuße für Ihr Spiel führt.

-vanillaGFX: Diese Option schaltet die Überprüfung des Treiberherstellers aus und kann sehr nützlich sein, um Ihr Spiel "wie es ist" zu testen.

Es ist erwähnenswert, dass die <filename> in -output und -debugoutput auf Wunsch dieselbe sein kann, da die Datei bei jedem Schreibvorgang geöffnet und geschlossen wird.

Gehen Sie nun zu dem Ort, an dem Sie die Verknüpfung zur ausführbaren Datei des Spiels erstellt haben, klicken Sie mit der rechten Maustaste darauf und wählen Sie Eigenschaften.

Daraufhin öffnet sich ein Fenster, das dem obigen ähnlich ist. Hier können Sie in den Zielpfad Ihre Befehlszeile einfügen. Blättern Sie einfach bis zum Ende des angegebenen Pfades, fügen Sie ein Leerzeichen und dann die Parameterzeichenfolge ein. Um zum Beispiel eine Debug-Datei zu erstellen, müssten Sie etwas wie folgt eingeben

D:\Users\Me\GameMaker\CustomLoadBar-Default-1.0.0.0\CustomLoadBar.exe -debugoutput debug.txt

Dadurch wird nun eine Datei mit dem Namen "debug.txt" am selben Ort wie die ausführbare Datei (nicht der Ort der Verknüpfung, sondern der Ort, an dem das Spiel installiert ist) mit Debug- und compiler Informationen aus Ihrem fertigen Spiel erstellt. Klicken Sie auf "OK" und verwenden Sie dann die Verknüpfung, um Ihr Spiel zu starten, damit es diese Parameter verwendet und Ihnen bei debugging hilft.

Weitere Informationen zum Debuggen Ihres Spiels finden Sie im Abschnitt Debugging des Handbuchs.

A/B-Tests über die Befehlszeile

Die YoYo runner profitiert auch von einem speziellen Befehlszeilenparameter:

-game <filename>Lädt die angegebene Spieldatei

Warum sollten Sie dies verwenden? Nun, normalerweise würde man das nicht tun, aber wenn man mehr als eine Version seines Spiels gleichzeitig laufen lassen möchte, kann man das auf diese Weise tun, was vergleichende A/B-Tests viel einfacher macht. Mit dieser Methode können Sie sich auf Tests mit anderen konzentrieren oder als Designer ein besseres Gefühl für die Dinge bekommen, indem Sie Seite-an-Seite-Tests durchführen.

Um dies einzurichten, müssen Sie vorher einige Dinge wissen. Zunächst einmal benötigen Sie den Pfad zu Runner selbst. Dieser befindet sich im Ordner " Programmdaten" Ihres Windows Betriebssystems (dieser ist möglicherweise standardmäßig ausgeblendet, so dass Sie die Anzeigeeigenschaften ändern müssen, um ihn zu sehen) und hat ein typisches Format wie dieses:

C:\ProgramData\GameMaker\Cache\runtimes\<runtime_version>\windows\Runner.exe

Beachten Sie, dass der Ordner runtime_version von der runtimes abhängt, die Sie derzeit installiert haben. Sie benötigen auch den Pfad zur Datei <yourgame>.win, da wir die runner auf diese Datei verweisen werden. Am einfachsten ist es, wenn Sie im Temp-Ordner (in der Regel unter %localappdata%\Temp\GameMakerStudio\<game_name>\) nach *.win suchen. Dann werden Ihnen alle verfügbaren Dateien angezeigt und Sie können einfach diejenige auswählen, von der Sie den vollständigen Pfad benötigen.

Wir könnten nun eine Verknüpfung zu runner erstellen, wie wir es für die ausführbare Datei getan haben, und dann die Parameter nach Bedarf ändern, aber das bedeutet, dass wir die Datei zwischen den Tests öffnen und bearbeiten müssen, was den Zweck dieser Funktion für Side-by-Side-Tests zunichte macht. Stattdessen werden wir ein paar kleine "Batch"-Dateien erstellen, um dies zu erledigen.

Erstellen Sie dazu zunächst ein neues Textdokument auf Ihrem Desktop und öffnen Sie es in Notepad (oder einem anderen Texteditor). Fügen Sie das Folgende hinzu:

START C:\ProgramData\GameMaker\Cache\runtimes\<runtime_version>\windows\Runner.exe -game <LOCAL TEMP PATH>\<GAMENAME>.win

Sie müssen den temporären Pfad so einstellen, dass er auf Ihre Spieldatei *.win verweist, und diese dann als "GM_Test_1.bat" auf Ihrem Desktop speichern (beachten Sie, dass Notepad automatisch als Datei *.txt speichert, es sei denn, Sie wählen "alle Dateien" in den Optionen des Speicherdialogs). Ein Beispiel dafür, wie Ihr endgültiger Befehl aussehen sollte, wäre:

START C:\ProgramData\GameMaker\Cache\runtimes\runtime-9.9.1.199\windows\Runner.exe -game D:\Users\Mark\AppData\Local\Temp\GameMaker\AlphaDog_Reboot\AlphaDog_Reboot.win

Erstellen Sie nun eine weitere Textdatei und fügen Sie dieselbe Zeile ein, nur dass Sie diesmal den Pfad für die zweite Version des Spiels angeben, die Sie nebeneinander testen möchten, und speichern Sie diese unter "GM_Test_2.bat". Sie können auf doppelklicken, um die verschiedenen Versionen Ihres Spiels aufzurufen, und da sie als Batch-Dateien vorliegen, können Sie sie leicht wieder öffnen, um sie zu bearbeiten und das getestete Spiel zu ändern.
